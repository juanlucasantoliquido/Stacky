## E2 — Verificación SLSA-3 contra releases reales

**Releases inspeccionados:** v0.5.5 → v0.8.1 (los disponibles al 2026-06-30).

**Artefactos de seguridad por release:**
- Binarios firmados con Sigstore/cosign (keyless) — archivos `.bundle` incluidos en cada release
- SHA-256 checksums para todos los binarios
- SBOM en formato SPDX 2.3
- Provenance via GitHub Actions `attest-build-provenance`
- Verificación: `gh attestation verify <file> --repo DeusData/codebase-memory-mcp`

**Claim del README:** "SLSA Level 3 — cryptographic build provenance generated by GitHub Actions; verify with `gh attestation verify <file> --repo DeusData/codebase-memory-mcp`"

**Verificación ejecutada:**  
El comando `gh attestation verify` usa el mecanismo de attestation de GitHub Actions + Sigstore. Este mecanismo ES compatible con SLSA Level 3 (prueba que el binario fue built por el workflow de este repo, no por actores externos). Sin embargo:
- No hay **certificación formal de tercero** (ej. SLSA Inspector externo)
- No hay evidencia de **builds reproducibles** (reconstruir el binario desde fuente da el mismo hash)

**Conclusión D3:** DUDOSO — cosign bundles + GitHub attestation presentes (prácticas SLSA-3 implementadas, claim explícito), pero sin cert formal de tercero ni reproducibilidad verificada. Riesgo supply-chain MEDIO.

## E4 — Solapamiento con índices de Stacky

### vs REPO_MAP.md
- `REPO_MAP.md` es un archivo estático Markdown generado externamente (el script `repo_map.py` NO existe en el repo — refutado).
- NO es queryable en runtime; no tiene API ni MCP.
- `codebase-memory-mcp` ofrece grafo dinámico queryable, 14 tools, sub-ms. **No solapa: complementario.**

### vs MCP interno de Stacky (`stacky_get_skill`)
- El MCP interno (`services/stacky_mcp.py`) expone `stacky_get_skill`: sirve **skills** de Stacky (procedimientos), NO estructura de código del proyecto del operador.
- `codebase-memory-mcp` indexa el **código fuente** del proyecto (Python/C#/TS/SQL) → queries de estructura de código.
- **No solapa:** dominios distintos (skills de Stacky vs código del proyecto). Coexistencia confirmada vía claves distintas en `mcpServers`.

**Conclusión D4:** APROBADO — aporte complementario claro, sin solapamiento real con ningún índice existente.
