Las mejoras llegan desactivadas y sin interrumpir la operación: cada equipo las activa a su ritmo, cada una pasa controles de calidad automáticos antes de liberarse, y el aislamiento por equipo garantiza que lo que pasa en uno nunca afecte a otro. Esto sostiene la inversión en el equipo experto.

Las versiones no se liberan a todos de golpe: a cada equipo se le entrega solo lo que va necesitando. La liberación general llega por necesidad o cuando se acumulan suficientes mejoras con la madurez probada; recién ahí pasan al resto de los equipos, aunque no lo pidan.

Generación de trabajo y motores de IA

90+ capacidades · 8 áreas funcionales

🧠 Agentes y generación de trabajo

  • Business Agent — brief/conversación de negocio → Épica HTML estructurada (bloques RF-XXX, actores, reglas, prioridades).
  • Auto-publicación de épica — POST /tickets/epics/from-brief; idempotente por marcador; sin aprobación (excepción pedida).
  • Vista previa de épica — EpicPayloadPreview + GET /epic-preview antes de publicar.
  • Portafolio de épicas — un mismo brief puede derivar N épicas.
  • Descomposición Épica→hijos — Features/Tasks con plan idempotente y anti-drift por fingerprint.
  • Functional Agent — análisis de cobertura CUBRE/GAP/NUEVA + plan de pruebas.
  • Technical Agent — traducción funcional→técnico (alcance, plan, unidades de trabajo).
  • Developer Agent — implementación: cambios en repo + comentario de evidencia.
  • QA Agent — veredicto PASS/FAIL + pipeline QA/UAT (Playwright).
  • Debug / PR-Review / Critic / Custom — agentes especializados adicionales.
  • Saneamiento de épica — sanitize determinista idempotente + warnings estructurales/catálogo.

🔌 Runtimes y motores de IA

  • Multi-runtime — Claude Code CLI, Codex CLI, GitHub Copilot, VS Code bridge.
  • Regla de no-fallback — si el runtime CLI falla, error real (nunca cae a Copilot en silencio).
  • LLM router — elige modelo por agente + complejidad + backend.
  • Selector adaptativo — modelo y nivel de esfuerzo según el confidence del caso.
  • Cap de modelos (clamp_model) — Opus 4.8 solo en brief→épica; Sonnet 4.6 default; Haiku en QA.
  • Niveles de effort — low / medium / high / xhigh / max, con clamp por modelo.
  • Difficulty routing — downgrade (S) / upgrade (L–XL) dentro del cap.
  • Dedup de prompts — invocation_block como fuente única en los 3 runtimes.
  • Paridad de streams — visibilidad stdout/stderr homogénea entre runtimes.
  • Disciplina de procesos — reúso de procesos por default.
  • Pre-ejecución especulativa — latencia-cero (FA-36).
  • Timeouts finitos + skip-permissions — 1800s default, evita procesos zombie.
  • Cap de concurrencia — run_slots + preflight de run.

🎯 Grounding, contexto y anti-alucinación

  • Grounding en catálogo — process_catalog + technical_master del cliente.
  • RAG TF-IDF — recupera el top-K del catálogo por similitud.
  • Confidence scoring — nivel de certeza por épica, con rescate.
  • Catálogo editable por UI — vía client_profile, sin tocar archivos.
  • Cross-ticket retrieval (FA-01) — contexto de tickets relacionados (TF-IDF).
  • Contexto vivo de BD (FA-02) — inyección de datos reales.
  • Búsqueda semántica de código (FA-03).
  • Git context (FA-05) — file-context y context-block del repo.
  • Mapa de cobertura de tests (FA-06).
  • Glosario / RIDIOMA (FA-09 / FA-15) — auto-construido y auto-inyectado.
  • Constraints de proyecto (FA-08).
  • Observatorio de grounding — pasivo + sugeridor de diccionario sin alucinar.

🛡️ Calidad, gates y verificación

  • Gate de Épica — severidad estructural + golden catalog diff.
  • Gate de Task — validación determinista.
  • Gate de regresión golden — positivo + negativo.
  • Bucle de convergencia — itera y mejora hasta el estándar (máx. configurable).
  • Pase correctivo (epic_repair) — repara output inválido.
  • Golden-set de extracción — funciones puras + ratchet meta + centinela de no-determinismo.
  • Contract Validator (N1) + Structured Output Renderer (N2).
  • Drift detection (FA-16).
  • Auto-typecheck del Developer (FA-17) + auto-SELECTs (FA-18).
  • Few-shot del mejor output (FA-12).
  • Conformance de runtimes — tests de paridad.
  • needs_review ruidoso — marca lo que no valida para revisión humana.

Operación, plataforma y gobernanza

90+ capacidades · 8 áreas funcionales

📊 Operación, observabilidad y daemons

  • Panel de Salud Operativa — triage en buckets: needs_review / failed / caros / zombies.
  • Historial de ejecuciones — inmutable, prompt + output por ticket.
  • Desatascador — detecta y remedia tasks atascadas; file picker universal; cap configurable.
  • Cierre automático de runs — output_watcher + manifest_watcher.
  • Orphan reaper + stale recovery — recupera runs colgados/zombie.
  • Cancelación de runs — POST /executions/{id}/cancel.
  • Diagnósticos / métricas / health.
  • KPIs de calidad + evals scheduler + eval history.
  • Cost preview (FA-33) + cache por hash (FA-31) + re-exec por diff (FA-32).
  • Digest daemon + memory review daemon.
  • Logs en vivo (SSE) + system logs (export CSV/JSON, stats, purge).
  • Backup semanal de la base de datos.

⚙️ Plataforma, configuración y gobernanza

  • Multi-proyecto — switch en 1 click, aislamiento por proyecto.
  • Todo configurable por UI — HarnessFlagsPanel: ~142 flags en ~13 categorías, búsqueda, defaults visibles.
  • Perfiles del arnés — off / safe / full + client profile + global config.
  • Human-in-the-loop — la persona aprueba (excepción: brief→épica auto-publica).
  • Mono-operador — sin login real; identidad por header X-User-Email.
  • Gateway con token — X-Stacky-Agent-Token para completar runs.
  • PII auto-masking (FA-37).
  • Cadena de auditoría HMAC (FA-39).
  • Derecho al olvido / GDPR (FA-40) + control de egress (FA-41).
  • Deploy congelado por máquina — harness_defaults.env horneado.
  • Pipeline de planes — proponer → criticar → implementar → supervisar (ledger versionado).
  • Power-user — macros DSL (FA-51), agent forking (FA-50), exploración paralela (FA-49), refinamiento multi-paso (FA-48).
  • Agent Packs — recetas guiadas + context chaining + re-run con edición.
