1. TRADUCCIÓN FUNCIONAL → TÉCNICA

Requerimiento funcional (resumen)

El CSV de interfaz v4.1 define los campos PGCANAL (varchar 6) y PGBONIF (decimal 13,2) para el par IN_PAGO / RPAGOS. Actualmente PGCANAL no existe en ninguna de las dos tablas, y PGBONIF existe sólo en IN_PAGO pero no se propaga a RPAGOS.

Solución técnica propuesta

El proceso batch IncHost ejecuta Pagos.ProcesoPagos() → PagosDalc.InsertPagos(), que realiza un INSERT-SELECT masivo de IN_PAGO hacia RPAGOS. La solución es: (1) crear las columnas faltantes en BD vía DDL, y (2) agregar PGCANAL, PGBONIF a la lista de columnas del INSERT y al SELECT de ese método. PGBONIF ya existe en IN_PAGO (confirmado: el proyecto legacy Oracle ya lo lee desde esa tabla); sólo hay que incluirlo en el SELECT. PGCANAL se crea en ambas tablas y se lee desde IN_PAGO.

Flujo actual del sistema (cómo funciona HOY)

  1. Inchost.cs detecta el sub-proceso PAGOS (código 2) y crea una instancia de BusInchost.Pagos.
  2. Llama a Pagos.ProcesoPagos() → crea instancia de PagosDalc → llama a InsertPagos().
  3. InsertPagos() ejecuta el siguiente SQL:
    INSERT INTO RPAGOS (PGOBLIG, PGFECVAL, PGFECENT, PGHORAPAGO, PGNROCUOTA, PGFECSIS,
                   PGPERFIL, PGNMORA, PGHISTORICO, PGTIPO, PGTOTAL, PGCAPITAL,
                   PGINTCOR, PGIMPUEST, PGIDPAGO, PGGSCOB)
SELECT PGOBLIG, PGFECVAL, PGFECENT, '', 0, PGFECPROC, LOPERFIL, LONMORA,
       0, PGTIPO, PGTOTAL, PGCAPITAL, PGINTCOR, PGIMPUEST, '', 0
FROM in_pago
JOIN IN_RCYO ON OCOBLIG = PGOBLIG
LEFT JOIN RLOTE ON LOCOD = OCRAIZ
WHERE PGFECPROC = '{cGlobales.Hoy}'
    → PGCANAL y PGBONIF están ausentes: no se propagan a RPAGOS.
  4. Si el INSERT es exitoso → conn.ActualizarEstadistica("RPAGOS") + COMMIT.

Flujo propuesto (cómo debe funcionar DESPUÉS)

  1. (sin cambio) Inchost.cs → Pagos.ProcesoPagos() → PagosDalc.InsertPagos().
  2. [CAMBIO] InsertPagos() ejecuta el SQL con los nuevos campos:
    INSERT INTO RPAGOS (PGOBLIG, PGFECVAL, PGFECENT, PGHORAPAGO, PGNROCUOTA, PGFECSIS,
                   PGPERFIL, PGNMORA, PGHISTORICO, PGTIPO, PGTOTAL, PGCAPITAL,
                   PGINTCOR, PGIMPUEST, PGIDPAGO, PGGSCOB, PGCANAL, PGBONIF)
SELECT PGOBLIG, PGFECVAL, PGFECENT, '', 0, PGFECPROC, LOPERFIL, LONMORA,
       0, PGTIPO, PGTOTAL, PGCAPITAL, PGINTCOR, PGIMPUEST, '', 0, PGCANAL, PGBONIF
FROM in_pago
JOIN IN_RCYO ON OCOBLIG = PGOBLIG
LEFT JOIN RLOTE ON LOCOD = OCRAIZ
WHERE PGFECPROC = '{cGlobales.Hoy}'
  3. (sin cambio) Si exitoso → COMMIT.

2. ALCANCE DE CAMBIOS

DDL — ejecutar en aisbddev02 ANTES del deploy de código

  AND COLUMN_NAME IN ('PGCANAL','PGBONIF')

-- 2. DDL IN_PAGO (agregar PGCANAL):
ALTER TABLE IN_PAGO ADD PGCANAL varchar(6) NULL;

-- 3. DDL RPAGOS (agregar PGCANAL y PGBONIF):
ALTER TABLE RPAGOS ADD PGCANAL varchar(6) NULL;
ALTER TABLE RPAGOS ADD PGBONIF numeric(13,2) NULL;

Cambios en código — nivel de MÉTODO

trunk/Batch/Negocio/BusInchost/PagosDalc.cs — Capa: Batch / DAL

Cambio 1 — OBLIGATORIO: método InsertPagos() — línea 417

  • Antes: Lista de columnas del INSERT termina en PGIDPAGO, PGGSCOB. El SELECT correspondiente termina en '', 0. PGCANAL y PGBONIF ausentes.
  • Después: Agregar PGCANAL, PGBONIF al final de la lista de columnas del INSERT y al final del SELECT (leyéndolos directamente de in_pago, que los tiene tras el DDL).
  • Razón: Es el único punto del flujo activo donde los registros de IN_PAGO se copian a RPAGOS.

Cambio 2 — SECUNDARIO: struct TyPago — línea 8

  • Contexto: TyPago en este proyecto mapea campos de RPAGOS (no de IN_PAGO). Se usa en CruzarPagos(), que existe en Pagos.cs pero actualmente NO es invocado desde Inchost.cs. Agregar los campos al struct mantiene consistencia para uso futuro.
  • Después: Agregar public string PGCANAL; y public double PGBONIF; al struct TyPago.
⚠️ Nota — dos archivos PagosDalc.cs en el mismo namespace BusInchost:
1. trunk/Batch/Negocio/BusInchost/PagosDalc.cs — activo, compilado por Inchost.sln. Este es el archivo a modificar.
2. trunk/Batch/Negocio/BusInchost/BusInchost/PagosDalc.cs — proyecto legacy Oracle, ya tiene PGBONIF en su struct tyPago pero le falta PGCANAL. Si ese proyecto también se despliega de forma independiente, necesita el mismo tratamiento.

5. NOTAS PARA EL DESARROLLADOR

Convenciones del proyecto a respetar

  • Prefijo de columna PG — consistente con todos los campos existentes de la tabla de pagos.
  • Las columnas nuevas deben ser NULL en ambas tablas — no agregar NOT NULL para no romper interfaces que no las envíen.
  • No se usa cFormat.StToBD() en el INSERT-SELECT inline porque los valores provienen directamente de la tabla fuente (no de variables C#).

Precauciones

  • El DDL debe ejecutarse ANTES del deploy. Si el código se despliega primero, el INSERT fallará con "invalid column name PGCANAL" o "PGBONIF".
  • InsertaPagoBD() (línea 391) y CruzarPagos() existen en el código pero no son invocados por Inchost.cs en el flujo activo. No es necesario modificarlos para cumplir el ticket.
  • No existe validación de PGMEDIOCOBRO o PGFORMAPAGO en este módulo — PGCANAL no necesita participar en ninguna lógica de validación existente.
  • El archivo trunk/Batch/Negocio/BusInchost/BusInchost/PagosDalc.cs (legacy Oracle) también necesita PGCANAL si ese proyecto se compila/despliega de forma independiente. Confirmar con el equipo si aplica.

Patrón de referencia

El mismo patrón INSERT-SELECT masivo existe en otros módulos de IncHost. Agregar las columnas nuevas al final de la lista, manteniendo la correspondencia posicional entre INSERT y SELECT.
