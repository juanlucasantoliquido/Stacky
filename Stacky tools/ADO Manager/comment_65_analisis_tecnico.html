Agregar 4 nuevos controles de filtro en FrmAgenda.aspx (1 lista + 3 texto libre). Extender la firma de Consultar() y Siguiente() en las tres capas (RSFac → RSBus → RSDalc) con 4 parámetros nuevos. 🧪 Cómo probar (paso a paso)

  1. Abrir FrmAgenda.aspx — verificar que aparecen los 4 nuevos controles en la 3ª fila del filtro.
  2. Seleccionar Débito Automático = Sí → Buscar → resultado esperado: 0 filas (todos los registros son NULL en dev = "No").
  3. Seleccionar Débito Automático = No → Buscar → resultado esperado: todos los lotes agendados.
  4. Escribir un texto en Corredor (ej: parte de un valor existente en OGCORREDOR) → Buscar → solo lotes cuyo corredor contiene ese texto (insensible a mayúsculas).
  5. Escribir un nombre parcial en Nombre de Cliente (ej: "garcia") → Buscar → lotes cuyos clientes contienen ese texto en CLNOMBRE o CLAPELLIDO1.
  6. Escribir un número parcial en RUC (ej: "20") → Buscar → lotes cuyo CLNUMDOC contiene ese valor.
  7. Dejar todos los 4 campos vacíos → Buscar → mismo resultado que antes del desarrollo (sin restricción adicional).
  8. Combinar filtros (ej: NivelMora=C1 + NombreCliente="lopez") → Buscar → AND lógico correcto entre todos los criterios.

1. TRADUCCIÓN FUNCIONAL → TÉCNICA

Requerimiento funcional (resumen)

Agregar 4 campos de filtro en la Búsqueda Avanzada de la Agenda Personal: Débito Automático (lista), Corredor (texto libre), Nombre de Cliente (texto libre) y RUC (texto libre). Cada campo vacío no restringe la búsqueda. Todos operan en AND con los 7 filtros estándar y entre sí.

Solución técnica propuesta

Los 4 campos son opcionales. Las condiciones SQL se agregan únicamente cuando el campo tiene valor, respetando el patrón existente del método Consultar(). ROBLG y RCLIE ya participan en el FROM de ambas queries — no se requiere JOIN adicional. Para los campos texto se usa cFormat.StToBD(valor.ToUpper(), cFormat.TipoLike.Contiene) que produce '%VALOR%' — válido en SQL Server y Oracle sin bifurcación de motor.

Flujo actual (HOY)

  1. El usuario abre FrmAgenda.aspx con 2 filas de filtro (8 controles estándar).
  2. Al hacer clic en Buscar → FiltrarGrilla() → RSFac.Agenda.Consultar(CodUsuario, Desde, Hasta, Empresa, Perfil, Recomendacion, Region, NivelMora, TipoCliente, ModoPool) — 9+1 parámetros.
  3. RSFac → RSBus → RSDalc con la misma firma. RSDalc ejecuta 2 queries (AGENDAAUT con AGFNTREC=-1, AGENDAUSU con AGFNTREC>0).

Flujo propuesto (DESPUÉS)

  1. (sin cambio) El usuario abre FrmAgenda.aspx — ahora con 3 filas de filtro (11 controles + 2 estándar que ya no tienen fila).
  2. [CAMBIO] Al hacer clic en Buscar → FiltrarGrilla() → RSFac.Agenda.Consultar(..., DebitoAuto, Corredor, NombreCliente, RUC, ModoPool) — 13+1 parámetros.
  3. RSFac → RSBus → RSDalc con firma extendida. RSDalc agrega las 4 condiciones SQL opcionales en ambas queries.

5. NOTAS PARA EL DESARROLLADOR

Convención SQL del proyecto

Para condiciones WHERE en métodos que usan StringBuilder (como Consultar() y Siguiente() en RSDalc/Agenda.cs), usar sSql.Append($@"...") — interpolación de cadenas verbatim. Para LIKE "contiene" insensible a mayúsculas: cFormat.StToBD(valor.ToUpper(), cFormat.TipoLike.Contiene) produce '%VALOR%' y escapa comillas simples internamente (protección SQL injection).

NO usar concatenación SQL Server-específica del tipo LIKE '%' + UPPER({0}) + '%'.

Patrón completo para condición opcional con LIKE:

if (!string.IsNullOrWhiteSpace(Campo))
    sSql.Append($@" AND UPPER(COLUMNA) LIKE {cFormat.StToBD(Campo.ToUpper(), cFormat.TipoLike.Contiene)} ");

Referencias en el codebase:

  • RSDalc/Cliente.cs ~567: sSql.AppendFormat(" AND CLNUMDOC LIKE{0} ", cFormat.StToBD(Documento, cFormat.TipoLike.Contiene))
  • RSDalc/Usuario.cs ~132: sSql += $@" CONVERT(VARCHAR, CAST(UFFECBAJA AS DATE), 103) FECHABAJAUSUARIO, " (patrón += en variables string, no StringBuilder)

Precauciones

  • Las condiciones SQL nuevas deben replicarse en ambas queries de Consultar() (sSql → AGENDAAUT y sSql2 → AGENDAUSU). Es fácil olvidar una.
  • Siguiente() en RSDalc usa concatenación directa ('" + value + "'") para los filtros existentes — mantener consistencia con ese estilo para los 4 nuevos en ese método.
  • OGCORREDOR existe en BD dev (VARCHAR(20)) pero no tiene script ALTER en el repo. El Developer no necesita agregar DDL, pero debe documentar esta deuda técnica en un comentario de código.
  • No modificar la lógica de ordenamiento (ORDER BY LODEUDA * RCAMBIO) ni los campos del SELECT — el ticket solo agrega condiciones WHERE.
